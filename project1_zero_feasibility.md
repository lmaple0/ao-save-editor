# “改之理：零式”UI 与功能借鉴可行性（只读调研）

## 调研边界与证据

- 观察对象：`Project1.exe` 0.21 Beta，仅切换页面查看，不执行保存、全开手册或写入存档。
- 程序是 32 位 Visual Basic 6 应用：静态字符串包含 `MSVBVM60.DLL`、`TABCTL32.OCX`、`COMDLG32.OCX`，以及原工程路径 `Project1.vbp`。
- 程序没有数字签名；本机样本 SHA-256 为 `EDB6100E5AB0A61C16309FBEAFF4796133449E8CDF0AD488CE16D08D50145857`。
- 随附 `Project1.exe.le.config` 指定 `zh-TW`、台北时区和注册表重定向；文本资源为 CP950 编码。
- 静态字符串能确认 `ReadCurrentSave`、`GetFishData`、`SaveFishData`、`savedata.dat` 等函数/路径线索，但没有源码，不能把它的偏移直接当作 Ao NISA PC 的可靠依据。

## 值得借鉴的 UI 结构

1. 左侧存档列表、右侧当前存档编辑区的主从布局，适合批量浏览多个存档。
2. 人物、装备、回路、队伍放在同一工作区，修改上下文集中。
3. 料理、钓鱼、怪物手册采用列表加勾选状态，能表达比“已登记/未登记”更细的状态。
4. 所有手册修改集中到明确的保存按钮，降低误写概率。

不建议照搬固定像素、信息密度过高的 VB6 布局。Ao Save Editor 应保留现有多语言、搜索过滤、诊断和分栏详情结构，只吸收其工作流。

## 本机 NISA 存档验证结论

默认目录 `C:\Users\<用户名>\Saved Games\FALCOM\Ao` 中最新有效存档显示：246 个已登记怪物的第四字节均为 `FF`，第三字节为 `7F` 或 `FF`；第一字节会随历史存档增加，第二字节也存在正常变化。当前程序使用旧欢乐百世版 `08 FE FF FF` 做严格相等判断，因此会把 NISA 的完整调查记录全部误报为“部分资料”。

安全的只读完整判定是：第三字节低 7 位全开，且第四字节为 `FF`。旧 payload 仍可作为兼容值识别。写入方面暂不应把第一、第二字节强制覆盖为旧版常量。

## 建议实施顺序

### P0：批量存档入口

- 用 Windows Known Folder API 获取 Saved Games，再拼接 `FALCOM\Ao`；找不到时回退到用户选择目录。
- 递归列出 `dataNNNN\savedata.dat`，显示槽号、修改时间、章节/游戏时间（能可靠解析的字段）、校验状态。
- 默认只读预览；编辑仍以单个已选存档为事务，保存前继续自动备份。

### P1：人物装备与回路只读页

- 先显示原始槽位、解析后的物品名和合法性诊断。
- 用多个自然存档和游戏内换装前后差分确认偏移，再开放下拉写入。
- 下拉列表按角色、装备类别、核心回路限制过滤，未知值可见且不自动替换。

### P1/P2：钓鱼手册

- 先从 Ao NISA 存档与本机游戏数据定位鱼种记录结构。
- 第一阶段只读展示鱼种、是否钓到、最大尺寸和次数；完成差分验证后再提供单项/批量修改。
- Project1 的鱼名、鱼饵与列表交互只作为 UI 参考，不作为 NISA 偏移证据。

### P2：怪物字段级研究

- 针对同一敌人做“初遇、击破、使用战斗探测器、获得掉落、重复击破”的受控存档差分。
- 将四字节拆为命名位字段前，保留原始十六进制诊断视图。
- 只有被两类证据（游戏内行为差分与代码/资源）同时支持的位才开放写入。
